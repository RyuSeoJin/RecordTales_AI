## 1. 파일 및 폴더 명명

### 1.1. 파일명

- **kebab-case** 사용. 소문자 + 하이픈.
- 확장자는 `.md`.
- 예: `project-identity.md`, `quests-panel.md`, `guild-member-hiring.md`.
- 금지: `ProjectIdentity.md`, `project_identity.md`, `프로젝트정체성.md`.

### 1.2. 폴더명

- **kebab-case** 사용.
- 예: `project-roadmap/`, `project-plan/approved/`, `project-plan/draft/`.

## 2. 문서 내 번호 표기

### 2.1. 모든 번호 뒤에는 `.`을 붙인다 (절대 규칙)

1단계부터 모든 레벨에서 번호 뒤에 점을 붙입니다.

| 레벨 | 올바른 표기 | 잘못된 표기 |
|------|------------|-----------|
| 1단계 | `1.`, `2.`, `3.` | `1`, `1)`, `1장` |
| 2단계 | `1.1.`, `2.3.` | `1.1`, `2.3` (뒤 점 없음) |
| 3단계 | `1.1.1.`, `2.3.4.` | `1.1.1`, `2-3-4` |

### 2.2. 적용 범위

다음 위치 **모두**에 적용:

- 목차(TOC) 항목
- 섹션 헤더 (`##`, `###`, `####`)
- 본문 내 다른 섹션 참조 (예: "2.1.에서 설명한 것처럼")
- 번호 매긴 리스트 (예: `1.`, `2.`, `3.`)

### 2.3. 예시

```markdown
## 1. 섹션 제목

### 1.1. 하위 섹션

#### 1.1.1. 세부 항목

본문에서 언급 시: "2.3.의 규칙에 따라…"
```

## 3. 다이어그램 정책

| 용도 | 도구 |
|------|-----|
| 상태 전이도, ERD, 플로우차트, 시퀀스 | **Mermaid** |
| 레이아웃, 와이어프레임, 코너 배치 | **ASCII 아트** |
| 작은 표, 매트릭스 | **마크다운 테이블** |

복잡한 관계나 상태 전이는 반드시 Mermaid로 표현합니다. 단순 레이아웃은 ASCII가 더 직관적일 수 있습니다.

## 4. 언어 표기

### 4.1. 기본 원칙

- **본문**: 한국어
- **기술 용어·고유 명사**: 영문 병기
- **코드·파일명·CSS 클래스**: 원문 그대로 (영문)

### 4.2. 병기 방식

- 한글(영문) 순서: "반복 마스터(recurring master)", "시작일(`startDate`)"
- 최초 등장 시에만 병기. 같은 문서 내 이후 등장은 한글 또는 영문 중 하나로 일관되게.

## 5. 레이아웃 영역 용어 (확정)

6개 영역 용어는 **모든 문서**에서 아래 표를 따릅니다. 신규 영역 추가 시 이 표를 먼저 갱신합니다.

| 영역 | 확정 용어 | 한글 |
|------|---------|------|
| 좌상 | `daily-status-panel` | 데일리 스테이터스 패널 |
| 좌하 | `audio-player-panel` | 오디오 플레이어 패널 |
| 우상 | `schedule-nav-bar` | 스케줄 내비게이션 바 |
| 우하 | `guild-menu-panel` | 길드 메뉴 패널 |
| 중앙 | `focus-main-display` | 포커스 메인 디스플레이 |

상세 정의 및 각 영역의 기능은 `project-dictionary.md` 참조.

## 6. 문서 필수 요소

모든 문서는 아래 요소를 포함합니다.

### 6.1. 문서 상단

- 제목 (`#` 1개)
- 문서 성격을 설명하는 인용구(`>`) 블록

### 6.2. 목차

- 1단계(`##`)까지는 필수 표기, 2단계(`###`)는 필요시 포함
- 번호 표기 규칙(2.1.) 준수

### 6.3. 문서 하단 — 업데이트 이력

모든 문서 최하단에 아래 형식의 표:

```markdown
## 📝 업데이트 이력

| 날짜 | 변경 내용 |
|------|----------|
| YYYY-MM-DD | 초안 작성 |
```

### 6.4. 관련 문서 섹션

마지막에서 두 번째 섹션(업데이트 이력 바로 위)에 다른 문서로의 인덱스를 표로 제공합니다.

### 6.5. 문서별 특수 원칙 — 섹션 0에 배치

어떤 문서 유형(예: 용어 사전, 시스템 스펙)은 이 가이드의 일반 규칙 외에 **고유한 작성 원칙**을 필요로 합니다. 이런 원칙은 **반드시 해당 문서의 최상단 섹션 `0.`에 배치**합니다.

**이유**: AI 에이전트와 작성자가 문서를 수정할 때 실제로 읽는 파일은 그 문서 자체입니다. 특수 원칙을 다른 문서에 두면 간접 참조가 되어 빠뜨릴 위험이 커집니다. 원칙이 문서 자신의 첫 섹션에 있어야 수정 전에 반드시 눈에 들어옵니다.

**규칙**:

- 섹션 번호는 `0.`을 사용 (본문 `1.`과 구분되는 "전제 섹션")
- 섹션 제목 예: `0. 이 문서를 작성·수정하기 전에 반드시 읽을 것`
- 섹션 내부 구조는 문서 성격에 맞게 자유롭게 구성 (원칙, 결정 규칙, 검증 체크리스트, 예시 등)
- 섹션 첫 문장에 **"AI 에이전트 및 작성자는 이 섹션을 먼저 읽고 내용을 숙지한 뒤에만 이후 섹션을 수정한다"** 같은 명시적 지시 문구 포함

**적용 대상 예시**:

- `project-dictionary.md` — UI 사전의 진입 경로 원칙
- 향후 복잡한 시스템 스펙 문서 — 스펙 변경 절차, 하위 호환 원칙 등

## 7. 태그 규칙

현재 HTML 프로토타입 상태와 리팩터 목표를 구분하기 위해 다음 태그를 사용합니다.

| 태그 | 의미 |
|------|-----|
| `[현재]` | 현재 HTML 프로토타입(`guild-master-desktop.html`)에 구현된 상태 |
| `[목표]` | 리팩터 완료 또는 향후 구현 목표 |
| `[공통]` | 현재와 목표 모두에 해당 |

본문 예시: "인스턴스는 [목표] 실제 물리 객체로 저장됩니다. [현재]는 파생 계산 방식."

## 9. 문서 톤

### 9.1. 기본 스타일

- 기술적이되 읽기 편한 혼합 스타일
- 한국어 본문, 용어는 영문 병기
- 섹션 번호 + 표 + 체크리스트 적극 활용

### 9.2. 절대적 규칙 표현

변경 불가능한 규칙·불변 조건을 설명할 때는 **"반드시"**, **"절대"**, **"금지"** 등의 강한 표현을 사용해 여지가 없음을 명확히 합니다.

예:
- "시작일(`startDate`)은 **절대** 변경되지 않는다."
- "마감일 오프셋은 마스터 생성 후 **반드시** 고정된다."
